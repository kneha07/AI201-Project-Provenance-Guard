# Provenance Guard

A Flask API for attributing content as human-written or AI-generated, with transparent labeling and a creator appeals workflow.

📹 **[Demo Video](https://www.youtube.com/watch?v=cFogACuIluo)**

---

## 1. Project Overview

Provenance Guard analyzes text submissions and returns a confidence score indicating whether the content was likely written by a human or generated by an AI system. It combines two independent detection signals — an LLM-based probability estimate via the Groq API and a set of local stylometric heuristics — into a single weighted confidence score. Based on that score, the system assigns one of three attribution labels: `likely_human`, `uncertain`, or `likely_ai`. Every submission is logged to a SQLite database. Creators who dispute a verdict can file an appeal, which queues the submission for human moderator review.

---

## 2. Architecture Overview

### Submission flow

A caller sends a POST request to `/submit` with a `text` field and a `creator_id`. The server validates that both fields are present and non-empty. It then runs the text through two independent detection signals in sequence: first the LLM signal (a Groq API call to `llama-3.3-70b-versatile`), then the stylometric signal (three local heuristics computed from the raw text). The two scores are combined using a 60/40 weighted average. The resulting confidence value determines the attribution label and the human-readable transparency label text. The full result is written to a SQLite database and returned as JSON.

### Appeal flow

A caller sends a POST request to `/appeal` with a `content_id` (returned by `/submit`) and a `creator_reasoning` string. The server looks up the content ID in the database, updates its `status` to `under_review`, and stores the reasoning text. A human moderator can then query `GET /log` to see all pending appeals alongside the original signal scores.

---

## 3. Detection Signals

### Signal 1: LLM via Groq (weight: 60%)

**What it measures:** The probability that a piece of text was generated by an AI system, as assessed by `llama-3.3-70b-versatile`. The model is prompted to return a single JSON field `ai_probability` as a float in [0.0, 1.0].

**Why chosen:** A large language model has internalized distributional patterns across millions of documents — it can detect subtle markers like overly balanced hedging, formulaic transition phrases, and the absence of personal voice that are difficult to encode as explicit rules.

**Blind spots:** The LLM can be fooled by AI text that has been lightly edited by a human, or by a human writer who deliberately mimics AI prose. It is also subject to Groq API availability; if the call fails, the function falls back to a neutral score of 0.5.

### Signal 2: Stylometric Heuristics (weight: 40%)

Three sub-metrics are computed locally from the raw text and averaged:

**Sentence length variance** — AI-generated text tends to produce sentences of similar length. Low variance across sentence lengths is treated as an AI signal. Normalized on a 0–40 scale; variance above 40 is fully human-like.

**Type-token ratio (TTR)** — Vocabulary diversity: unique word count divided by total word count. AI text often reuses the same word families, producing a lower TTR. A TTR below 0.4 scores as AI-like.

**Punctuation density** — Human writers use more varied punctuation (em dashes, semicolons, parentheses, ellipses). The density of these characters per 100 characters is measured; a density above 2 per 100 characters is treated as human-like.

**Why chosen:** Stylometrics are deterministic, require no external API, and add a completely independent axis of evidence that is harder to game simultaneously with the LLM signal.

**Blind spots:** Short texts (fewer than 2 sentences) produce unreliable variance. Formal academic or technical human writing may score as AI-like due to its disciplined prose conventions.

### Signal 3: Modal Verb Density (weight: 20%)

**What it measures:** Frequency of hedging modal verbs ("would," "could," "may," "might," "should," "can," "shall," "must," "ought") per 100 words. AI models systematically overuse these hedges to avoid making definitive claims.

**Why chosen:** Captures a specific lexical habit that is independent of both holistic LLM pattern recognition and structural stylometrics. Adds a third axis of evidence with a low false-positive rate on casual human writing.

**Blind spots:** Legal, compliance, and academic human writing legitimately uses many modals ("the system shall," "the applicant must"). This signal will over-score those genres.

## 4. Confidence Scoring

### How signals are combined

```
confidence = llm_score * 0.5 + stylometric_score * 0.3 + modal_score * 0.2
```

Three signals are combined in a weighted ensemble. The LLM signal receives 50% weight for its broad semantic pattern recognition. Stylometrics receive 30% as a fast, deterministic cross-check. Modal verb density receives 20% as a narrow, specific lexical tell. No single signal can flip the attribution alone — it must move the weighted average past a threshold with at least partial support from the others.

### Validation approach

Both signals were manually tested against a small corpus of clearly AI-generated text (produced with GPT-4) and clearly human-written text (personal essays, forum posts). Thresholds were adjusted until the corpus produced sensible labels. Full automated testing would require a labeled benchmark dataset, which is outside the scope of this prototype.

### Example submissions

**Example 1 — High-confidence AI-generated paragraph:**

Text: *"Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications. Furthermore, stakeholders across various sectors must collaborate to ensure responsible deployment."*

| Signal | Score |
|--------|-------|
| llm_score | **0.92** (LLM assessed high AI probability) |
| stylometric_score | 0.6682 (uniform sentence lengths, no varied punctuation) |
| combined confidence | **0.82** |
| attribution | `likely_ai` |

**Example 2 — Lower-confidence human-written paragraph:**

Text: *"ok so i finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put WAY too much sodium in it and i was thirsty for like three hours after. my friend got the spicy version and said it was better. probably wont go back unless someone drags me there"*

| Signal | Score |
|--------|-------|
| llm_score | **0.23** (LLM assessed low AI probability) |
| stylometric_score | 0.6406 (high sentence variance but no varied punctuation) |
| combined confidence | **0.39** |
| attribution | `uncertain` |

The scores above are real API outputs captured from live testing. The AI paragraph scores 0.82 (likely_ai) while the casual human text scores 0.39 (uncertain, leaning human) — a difference of 0.43 across the confidence range, confirming the scoring produces meaningful variation between the two extremes.

## 5. Transparency Label

The `label` field in the `/submit` response contains one of three human-readable texts, shown here verbatim:

**Label variant 1 — `likely_ai`:**
```
Looks like AI-generated content (82% confidence). Our system detected patterns
commonly found in AI writing. If you used an AI tool to help write this, consider
adding a note so readers know. Think this is wrong? You can file an appeal.
```

**Label variant 2 — `likely_human`:**
```
Looks like human-written content (27% confidence). Our system found strong signs
that a person wrote this — things like natural variation in sentence length and
personal punctuation habits. Great news for your attribution!
```

**Label variant 3 — `uncertain`:**
```
We're not sure about this one (51% confidence). The writing has some qualities
our system associates with AI and some it associates with humans. That can happen
with carefully edited work or content that crosses genres. If you wrote this
yourself, you can file an appeal and tell us why.
```

Each label opens with the verdict in plain conversational language, includes the confidence percentage, explains *what the system actually looked at* in plain terms, and (where relevant) tells the creator what to do next. The uncertain variant explicitly names the appeal path without being alarming.

## 6. Rate Limiting

### Limits applied

- **10 requests per minute** per IP address
- **100 requests per day** per IP address

Rate limiting is applied only to `POST /submit` (the computationally and financially expensive endpoint that calls Groq). The `/appeal`, `/log`, and `/health` endpoints are not rate-limited.

### Reasoning

A legitimate human writer submitting their own work would rarely need to submit more than a few pieces per minute. The 10/min limit comfortably covers normal use while preventing a single caller from flooding the Groq API (which has its own rate limits and costs per token). The 100/day limit is generous for individual use but prevents automated bulk scanning of large content libraries without coordination.

## 8. Appeals Workflow

1. The creator submits text via `POST /submit` and receives a verdict they believe is incorrect.
2. They copy the `content_id` from the response.
3. They call `POST /appeal` with the `content_id` and a `creator_reasoning` string explaining their case (e.g., "I wrote this myself; the formal tone is because it is an academic summary").
4. The system sets the submission's `status` to `under_review` and stores the reasoning.
5. A human moderator queries `GET /log` to find all `under_review` entries. They review the original signal scores alongside the creator's explanation to decide whether the attribution was correct.
6. The moderator updates the record manually. (Automated resolution is not implemented in this prototype.)

### Appeal request example

```bash
curl -X POST http://localhost:5001/appeal \
  -H "Content-Type: application/json" \
  -d '{
    "content_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "creator_reasoning": "I wrote this essay myself. The stylistically even sentences are a result of careful editing, not AI generation."
  }'
```

### Appeal response example

```json
{
  "message": "Appeal submitted successfully. A human reviewer will examine your case.",
  "content_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "under_review"
}
```

## 9. Known Limitations

**Formal human writing misclassified as AI:** Academic essays, grant proposals, and technical documentation often have low sentence length variance and low punctuation density by convention. This causes the stylometric signal to score them as AI-like, and the LLM may follow suit if the topic is commonly covered in AI training corpora. The practical result is that a well-polished human essay may land in the `uncertain` band or even `likely_ai`, forcing the creator to file an appeal. A future mitigation would be to add a genre/domain classifier that adjusts thresholds for academic writing.

**Very short texts are unreliable:** Texts under 30 words do not provide enough signal for the stylometric heuristics. The system falls back to 0.5 for the variance sub-metric in these cases, which often produces an `uncertain` verdict regardless of the actual authorship.

## 10. Spec Reflection

**One way the spec helped:** The explicit requirement to separate detection into named signals with individual scores (rather than a single opaque score) forced a modular design where `llm_signal` and `stylometric_signal` are independent functions. This made it straightforward to test each in isolation and to expose both scores in the API response for transparency. Without that structural constraint from the spec, a simpler but less auditable implementation would have been tempting.

**One way implementation diverged from the spec:** The original plan called for the stylometric signal to include a passive voice rate metric. In practice, extracting passive voice reliably in Python without a dependency like spaCy requires complex regex patterns that proved fragile on diverse text. It was replaced with punctuation density, which is simpler to compute correctly and provides a comparable human-versus-AI signal. The spec was updated in planning.md to reflect this change.

## 11. AI Usage

**Instance 1 — Stylometric threshold calibration:** Claude was asked to suggest normalization boundaries for the sentence length variance metric. The initial implementation used a raw variance value without normalization, which caused the metric to saturate at 1.0 for any reasonably varied text. Claude suggested a 0–40 normalization range based on typical variance distributions in English prose. However, testing showed this cap was too aggressive — AI-generated text with only 3 sentences could easily produce variance above 40 and be incorrectly scored as human-like. The threshold was revised upward to 100 after observing actual variance values on test inputs.

**Instance 2 — Transparency label copy:** The first draft of the three label texts used technical language ("confidence interval", "probabilistic attribution"). Claude was prompted to rewrite them for a non-technical audience while preserving the verdict, the score, and the next step for the user. The AI draft used the word "assessment" throughout all three variants, making them feel repetitive. The final version was revised to vary the opening verb ("indicates" vs. "could not confidently determine") and to add the specific appeal call-to-action only in the uncertain variant, not in all three.

## 12. Stretch: Ensemble Detection (3 Signals)

The system uses three independent detection signals combined via a weighted ensemble.

### Signal 3: Modal Verb Density (weight: 20%)

**What it measures:** The frequency of hedging modal verbs — "would," "could," "may," "might," "should," "can," "shall," "must," "ought" — per 100 words. AI models systematically overuse these hedges to avoid making definitive claims; human writers use modals more sparingly and purposefully.

**Why chosen:** Modal overuse is a narrow, specific AI tell that is largely independent of the LLM and stylometric signals. The LLM signal captures holistic semantic coherence; stylometrics capture structural surface properties; modal density captures a single lexical habit.

**What it misses:** Legal documents, technical specifications, and academic writing legitimately use many modals. A human-written compliance document may score as AI-like on this signal alone.

**Output:** Float in [0.0, 1.0]. Density ≥ 6 modals per 100 words → 1.0 (fully AI-like). 0 modals → 0.0.

### Ensemble Weighting

| Signal | Weight | Rationale |
|--------|--------|-----------|
| LLM (Groq) | 50% | Broadest semantic pattern recognition; most reliable |
| Stylometric heuristics | 30% | Structural/surface properties; deterministic, no API dependency |
| Modal verb density | 20% | Specific lexical habit; low false-positive rate on casual text |

**Conflict resolution:** The weighted average handles signal conflicts implicitly — no single signal can flip the final attribution without support from at least one other signal. A signal scoring 1.0 while both others score 0.0 produces a combined score of at most its own weight, keeping the verdict in the uncertain range.

### Example response (all three signals visible)

```json
{
  "attribution": "likely_ai",
  "confidence": 0.82,
  "signals": {
    "llm_score": 0.92,
    "stylometric_score": 0.67,
    "modal_score": 0.50
  },
  "label": "AI-Generated Content\nOur analysis indicates this content was likely generated by an AI system (confidence: 82%). We encourage creators to disclose AI use."
}
```

## 13. Stretch: Analytics Dashboard

`GET /analytics` returns three metrics summarizing all submissions to date.

### Metrics

1. **Detection distribution** — count and ratio of each attribution label (`likely_ai`, `likely_human`, `uncertain`). Monitors whether the system is biased toward one verdict over time.
2. **Appeal rate** — fraction of submissions flagged as `under_review`. A rising appeal rate signals possible over-classification or miscalibration.
3. **Average confidence** — mean combined confidence score across all submissions. Persistently high or low averages indicate the scoring may be poorly calibrated for the platform's content.

### Sample response

```bash
curl http://localhost:5001/analytics
```

```json
{
    "appeal_rate": 0.125,
    "appeals_filed": 1,
    "average_confidence": 0.667,
```

## 14. Setup & Running Instructions

### Prerequisites

- Python 3.10+
- A [Groq API key](https://console.groq.com/) (free tier available)

### Installation

```bash
# Clone or copy the project directory
cd /Users/neha/ai201-project4-provenance-guard

# Create and activate a virtual environment
python3 -m venv .venv
source .venv/bin/activate

# Install dependencies
pip install -r requirements.txt
```

### Configuration

Create a `.env` file in the project root:

```
GROQ_API_KEY=your_groq_api_key_here
```

### Running the server

```bash
python3 app.py
```

The server starts on `http://localhost:5001` by default (port 5000 is reserved by macOS AirPlay Receiver). To use a different port:

```bash
PORT=8080 python3 app.py
```

### Example API calls

**Submit text for analysis:**
```bash
curl -X POST http://localhost:5001/submit \
  -H "Content-Type: application/json" \
  -d '{
    "text": "The implementation of artificial intelligence in healthcare presents numerous opportunities for improving patient outcomes while simultaneously raising important ethical considerations that must be carefully addressed.",
    "creator_id": "user_42"
  }'
```

**File an appeal:**
```bash
curl -X POST http://localhost:5001/appeal \
  -H "Content-Type: application/json" \
  -d '{
    "content_id": "<content_id from submit response>",
    "creator_reasoning": "I wrote this paragraph myself for a grant proposal."
  }'
```

**View audit log:**
```bash
curl http://localhost:5001/log
```

**Health check:**
```bash
curl http://localhost:5001/health
```
